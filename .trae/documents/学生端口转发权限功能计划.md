# 学生端口转发权限功能实现计划

## 1. 需求分析（Repo Research Conclusion）

现有项目是一个 JupyterHub 管理平台，前端为 Vue3 + Element Plus，后端为 Spring Boot + Spring Security。当前 admin 账号拥有所有功能栏的访问权限（容器管理、在线学生、文件管理、显卡分配、端口转发、定时任务、资源监控）。

**本次新增需求**：
- 新增 student1 ~ student60 共 60 个学生账号，统一密码 `test234`
- 学生登录后，仅能看到并使用"端口转发"栏，其他所有栏完全隐藏
- 学生页面的端口转发面板，学生选择框自动锁定为登录账号，不可切换；监控管理与监控日志也不应暴露给学生
- admin 保持原有全部权限，不得影响现有 UI/后端核心逻辑

**相关文件一览**（只读信息，未改动）：
- 后端：`backend/src/main/java/com/jupyterhub/service/AuthService.java`
- 后端：`backend/src/main/java/com/jupyterhub/controller/AuthController.java`
- 后端：`backend/src/main/java/com/jupyterhub/config/SecurityConfig.java`
- 前端：`frontend/src/router/index.js`
- 前端：`frontend/src/components/Layout.vue`
- 前端：`frontend/src/views/Login.vue`
- 前端：`frontend/src/views/PortForward.vue`

## 3. 详细修改步骤与内容

### 3.1 后端：AuthService — 学生账号识别与角色返回

**文件**：`backend/src/main/java/com/jupyterhub/service/AuthService.java`

**改动点**：
- 保留原有 admin 登录校验逻辑（不删除不修改）
- 在 `login(String username, String password)` 方法内新增分支：
  - 若 username 匹配 `student[1-9]` 或 `student[1-5][0-9]` 或 `student60`（可用正则 `"^student([1-9]|[1-5][0-9]|60)$"`），且 password 等于 `test234`，则校验通过并返回 STUDENT 角色
  - 其他用户名仍然走原 admin 校验路径（即 ADMIN）
- 新增返回对象字段（或扩展现有 Map/DTO）：加入 `role`（值为 `"ADMIN"` 或 `"STUDENT"`），并保留原返回的 `username` 与 `token` 等字段
- logger.info 输出中区分"管理员登录"/"学生登录"

**注意事项**：
- 不要动现有 admin 的 token 生成 / 密码比对逻辑
- 学生账号不需要数据库持久化，直接硬编码匹配即可（需求明确要求"写死"）

---

### 3.2 后端：AuthController — 返回结构增加 role 字段

**文件**：`backend/src/main/java/com/jupyterhub/controller/AuthController.java`

**改动点**：
- 在 `POST /auth/login` 的响应 JSON 中同步返回 `role` 字段
- 保持其他字段（token、username、code、message 等）不变
- 对学生账号，确保在 controller 层也记录一条 login 日志，便于排查

---

### 3.3 后端：SecurityConfig — 放行策略检查

**文件**：`backend/src/main/java/com/jupyterhub/config/SecurityConfig.java`

**改动点**：
- 只读核对 `/auth/**`、`/port-forward/**` 是否均已 `permitAll()` 或在鉴权白名单中
- **若未放行** `/port-forward/**`，则新增一行 `antMatchers("/port-forward/**").permitAll()`
- 其他 antMatchers 保持原样，绝不删除或变更

> 说明：由于目前前端是基于 token 走 axios 请求，/port-forward 的权限将以前端角色控制为主；后端侧保持与现有其他接口一致的访问策略即可。

---

### 3.4 前端：router/index.js — 角色感知路由守卫

**文件**：`frontend/src/router/index.js`

**改动点**：
- 路由表保持不变（所有 routes 保留），不删除任何已有路由
- 在全局 `beforeEach` 守卫中：
  - 登录用户为 `STUDENT` 时，强制默认跳转到 `/port-forward`；尝试访问非 `/port-forward` 的管理页时，重定向到 `/port-forward`
  - 登录用户为 `ADMIN` 时，维持现有默认跳转与全部页面访问
- 登录页 `/login` 保持公开

---

### 3.5 前端：Layout.vue — 侧边栏按角色过滤

**文件**：`frontend/src/components/Layout.vue`

**改动点**：
- 顶部"学生容器管理"等页面标题文本，**不删除**，但根据角色展示（或者直接使用 route.meta.title，复用现有实现）
- 侧边栏菜单数组，在遍历渲染前，按 `userInfo.role` 过滤：
  - `ADMIN`：原样显示所有栏
  - `STUDENT`：只显示"端口转发"一项
- 退出登录按钮保留；右上角"admin"标签替换为当前登录用户名（已有则不改）
- 不改动 Layout 的整体布局与样式（padding、背景、左侧栏宽度等保持一致）

---

### 3.6 前端：PortForward.vue — 学生模式 UI 裁剪

**文件**：`frontend/src/views/PortForward.vue`

**改动点**：
- 在 `onMounted` 或 computed 中读取 `userInfo.role`
- 当 `role === 'STUDENT'`：
  1. "学生"下拉框的值被强制设为登录用户名（student1 ~ student60），并且整个下拉框设置为 `disabled`，改为纯文本展示更直观（推荐使用 `el-input` 替换该字段，label 仍保留"学生"）
  2. 隐藏右侧"监控管理"与"监控日志"两个卡片（用 `v-if="isAdmin"` 包裹）
  3. "当前转发列表"表格的"删除"按钮保留，因为删除的是学生自己的转发条目
  4. 自动刷新开关保持默认开启
- 当 `role === 'ADMIN'`：一切维持原有表现（学生下拉可编辑、监控两个卡片可见）
- 不删除模板中任何现有代码，只用 `v-if` 进行展示控制

---

### 3.7 前端：Login.vue — 复用现有页面

**文件**：`frontend/src/views/Login.vue`

**改动点**：
- 如当前已有登录表单（截图显示已有），**无需改动**
- 确认登录后存储 `userInfo`（包含 username、role）与 `token` 到 localStorage / pinia / vuex（根据项目现有存储方案）
- 若未存储 `role` 字段，仅在这里补充一行 `role` 的写入，其他逻辑保持不变

## 4. 依赖与考虑事项

- **后端无新增依赖**：无需添加 pom.xml 依赖
- **前端无新增依赖**：无需 npm install 新包
- **数据层**：学生账号完全硬编码在 `AuthService.java` 中，不涉及数据库变更
- **端口转发脚本**：沿用之前的 `/opt/jupyterhub_prod/ip_res.sh`，无需改动脚本
- **Token 机制**：沿用现有 token 生成逻辑（确保学生与 admin token 均有效）
- **兼容性**：admin 登录体验保持 100% 不变；学生登录体验为新增路径，不会影响 admin

## 5. 风险处理与回退策略

| 风险 | 影响 | 处理 |
|------|------|------|
| 学生账号正则匹配有误（例如多匹配 `student100`） | 学生越权 / 无法登录 | 登录时在 AuthService 内部加断言与日志输出；正则明确为 `1-60` 区间，使用 `"^student([1-9]|[1-5][0-9]|60)$"` |
| 学生登录仍能访问其他栏接口 | 越权使用 | 前端 Layout 层面做 v-if 隐藏 + router 层面做守卫重定向，双保险；后端侧 `/port-forward/**` 保持不变但接口正常需要 token |
| role 字段前端读取失败（例如大小写不一致） | 角色逻辑不生效 | 统一使用大写 `ADMIN`/`STUDENT`，并在前端登录成功后 `console.log` 一次返回值，便于快速排查 |
| 代码编辑时误改动其他栏 | 污染其他功能 | 严格按本计划清单改动；改动前先用 Read 读取代码段，Edit 只替换指定片段，不在"非目标文件"中做编辑 |
| 热部署失败导致服务不可用 | 短暂服务中断 | 先在本地 `mvn compile` 检查编译错误；改动仅涉及 `AuthService/AuthController/PortForwardService` 三个文件，可直接重启后端 jar |

## 6. 验收标准（完成标准）

1. **admin 登录**：使用原账号密码登录，侧边栏显示全部栏目（容器管理、在线学生、文件管理、显卡分配、端口转发、定时任务、资源监控），所有栏目功能完全可用，与改动前一致。
2. **学生登录**：使用 `student23` / `test234`（任意 1~60 均有效）登录，侧边栏**仅显示"端口转发"**一项，其他栏在侧边栏不可见，且直接访问其 URL 会被路由守卫重定向到 `/port-forward`。
3. **学生版端口转发**：学生字段被锁定为登录用户名（不可修改），右侧监控两个面板隐藏；"添加转发"仍可使用（宿主机端口由学生自定义），"删除"仍可删除属于自己的转发条目。
4. **日志监控**：学生版页面不暴露监控日志与监控管理；admin 版保持原样。
5. **退出登录**：admin/学生均能点击"退出"回到登录页，且 token 被清除。
